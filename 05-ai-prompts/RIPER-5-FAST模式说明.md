# RIPER-5 FAST 模式说明

## 概述

**FAST 模式** 是 RIPER-5 工作流的**轻量级变体**，专门用于处理简单、低风险的编程任务。它跳过了 RIPER-5 中严格的五阶段流程，直接进入执行，以提高效率。

---

## 背景：为什么需要 FAST 模式？

### RIPER-5 的局限性

根据社区实践和用户反馈，RIPER-5 虽然在复杂任务中表现出色，但在处理简单任务时存在以下问题：

| 问题 | 描述 | 影响 |
|------|------|------|
| **流程过重** | 即使简单任务也要经历 Research → Innovate → Plan → Execute → Review | 浪费时间 |
| **过度复杂化** | 简单的 bug 修复被分解成多个文档和阶段 | 降低效率 |
| **认知负担** | 需要记住模式切换规则 | 增加心理负担 |
| **文档膨胀** | 每个简单任务都生成多个 md 文件 | 管理成本高 |

**用户反馈摘录：**
> "但是对于简单任务，RIPER-5 很容易让功能过度复杂化，降低工作效率。目前我的方案是不放在规则文档中，而是在需要使用时手动引用。"
> — 苏杭小站

---

## FAST 模式的定义

### FAST 是什么？

**FAST** 可能代表：
- **F**ast（快速）
- **A**ct（行动）
- **S**imple（简单）
- **T**ask（任务）

**核心思想：** 对于明确、低风险的简单任务，直接执行，跳过规划阶段。

---

## FAST vs RIPER-5 对比

| 特性 | RIPER-5 模式 | FAST 模式 |
|------|-------------|-----------|
| **适用场景** | 复杂功能、多文件改动、重构 | 单文件修改、简单 bug 修复、注释 |
| **阶段数** | 5 个（R-I-P-E-R） | 1-2 个（直接执行 + 可选审查） |
| **文档要求** | requirements.md + design.md + tasks.md | 无需或仅简单记录 |
| **用户授权** | 每阶段都需要明确授权 | 任务级别授权 |
| **可追溯性** | 完整的决策和实施记录 | 基础的修改记录 |
| **效率** | 较低，但可控性高 | 高，适合快速迭代 |

---

## FAST 模式的适用场景

### ✅ 适合使用 FAST 模式的任务

1. **单文件修改**
   - 修改变量名
   - 调整函数参数
   - 添加/删除几行代码

2. **简单 Bug 修复**
   - 明显的逻辑错误
   - 拼写错误修正
   - 简单的异常处理

3. **文档更新**
   - 添加注释
   - 更新 README
   - 修改文档说明

4. **格式化调整**
   - 代码格式化
   - 导入语句整理
   - 空行/缩进调整

5. **明确的微功能**
   - 添加简单的日志
   - 修改配置值
   - 添加简单的验证逻辑

### ❌ 不适合使用 FAST 模式的任务

1. **跨文件修改** → 使用 RIPER-5
2. **架构调整** → 使用 RIPER-5
3. **新功能开发** → 使用 RIPER-5
4. **性能优化** → 使用 RIPER-5
5. **重构** → 使用 RIPER-5
6. **涉及核心业务逻辑** → 使用 RIPER-5

## FAST 模式的实施方式

### 方式 1：手动激活（推荐）

**原则：** 不将 FAST 模式写入 AI 系统提示词，而是在需要时手动指定。

**操作示例：**
```
用户：这是一个简单任务，使用 FAST 模式直接修复这个拼写错误。

AI：[模式：FAST 执行] 正在修复 FileProxy.java 第 42 行的拼写错误...
```

**优点：**
- 避免默认滥用 FAST 模式
- 保持 RIPER-5 作为主要工作流
- 用户有完全控制权

---

### 方式 2：自动判断（需要 AI 具备判断能力）

**触发条件：**
```
如果 任务满足以下所有条件：
  - 单文件修改
  - 改动量 < 50 行
  - 不涉及架构变更
  - 风险评估为低

则 自动进入 FAST 模式
```

**实施建议：**
- 在系统提示词中添加判断逻辑
- 要求 AI 在进入 FAST 模式前明确说明原因
- 保留用户否决的权利

---

### 方式 3：混合模式（带简化计划）

**流程：**
```
1. 快速理解（5 秒）
2. 简化确认（一句话）
3. 直接执行
4. 可选审查
```

**示例：**
```
AI：[FAST 模式]
任务：在 UserService.java 添加 email 验证
风险评估：低，不影响现有功能
确认执行？(y/n)

用户：y

AI：正在执行...
```

## FAST 模式的最佳实践

### 1. 明确任务边界

在使用 FAST 模式前，确保任务确实符合"简单"标准：

```
✅ 简单任务的判断标准：
- 单个文件改动
- 改动 < 30 行代码
- 不改变接口或数据结构
- 不影响其他模块
- 可以在 5 分钟内完成
```

### 2. 保留最小化文档

即使是 FAST 模式，也建议记录：

```markdown
## FAST Task Log

### 2026-01-22: Fix typo in FileProxy.java
- **Type**: Bug Fix
- **File**: FileProxy.java:42
- **Change**: `transcode` → `transcodeTask`
- **Impact**: None (local variable)
- **Status**: ✅ Completed
```

### 3. 安全措施

即使使用 FAST 模式，也应：

- ✅ 要求 AI 执行前展示修改预览
- ✅ 保留版本控制（Git commit）
- ✅ 可选的代码审查步骤
- ❌ 不要禁用所有检查（盲目执行）

### 4. 降级机制

**如果在 FAST 执行中发现问题：**
```
遇到以下情况立即切换回 RIPER-5：
- 发现跨文件依赖
- 改动量超出预期
- 出现意想不到的副作用
- 用户要求停止
```

## 如何在 AI 助手中配置 FAST 模式

### 配置示例

```markdown
## 模式选择

### RIPER-5 模式（默认）
适用于：复杂任务、多文件改动、新功能开发

### FAST 模式（按需激活）
适用：单文件小修、简单 bug 修复、格式化

**用户可以通过以下方式激活 FAST 模式：**
1. 明确说明："使用 FAST 模式"
2. AI 自动判断（需符合简单任务标准）
3. 任务描述中包含"简单"、"快速"、"小修"等关键词

**FAST 模式执行规则：**
- 跳过 Research 和 Innovate 阶段
- 直接展示修改计划（1-3 句话）
- 等待用户确认后执行
- 完成后简要报告
```

## 总结

### FAST 模式的核心价值

| 价值 | 说明 |
|------|------|
| **效率提升** | 简单任务不需要繁琐流程 |
| **流程平衡** | RIPER-5 + FAST 覆盖所有场景 |
| **用户友好** | 减少不必要的认知负担 |
| **灵活性** | 按需激活，不强制使用 |

### 使用建议

1. **RIPER-5 优先**：将 RIPER-5 作为默认工作流
2. **FAST 按需**：只在明确简单的任务中使用 FAST
3. **边界清晰**：定义好简单任务的标准
4. **保留控制**：即使用 FAST 模式，也要有确认环节
5. **记录完整**：即使是 FAST 任务，也应在 Git 中记录

### 最终建议

> **FAST 模式不是 RIPER-5 的替代品，而是补充。**
> 它让开发者能够在保持控制的同时，快速处理简单任务。
